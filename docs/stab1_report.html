§1

Цель

Тест поставлен в ответ на аномальный выброс времени (~8100 с) в записи водопада с ESP32-платы (инцидент #FW-16). Требовалось установить: является ли дрейф усиления детектора источником нестабильности — или артефакт возникает на уровне тракта платы.

Для изоляции ESP32 детектор AtomSpectra подключён напрямую через FTDI-адаптер без участия платы-шлюза. Мониторинг выполнен в течение 9.6 ч с шагом окна 60 с (574 окна, из них 5 — baseline, 569 — измерительных).

Вывод: детектор стабилен. Максимальный дрейф пика за 9.6 ч — 0.023% при пороге ±2.0% (запас 85×). Источник аномалий #FW-16 — тракт ESP32.

§3

Протокол и методология

Бинарный протокол shproto. Детектор передаёт данные в собственном протоколе с escape-кодированием и CRC:

Фрейм: 0xFF 0xFE <data_escaped> 0xA5 Escape: байт 0xFD инвертирует следующий байт (~b & 0xFF) Контрольная: CRC16-Modbus (табличный алгоритм, 256 записей) CMD 0x01: порция гистограммы — offset 2 B + counts 4 B × N CMD 0x03: текстовый ответ (на команды -inf, -sta)

По команде -sta детектор начинает поток порций гистограммы. Скрипт накапливает инкрементальную гистограмму 8192 каналов. Каждые 60 с вычисляется разностная гистограмма (окно), затем центроид пика Cs-137 в диапазоне каналов 1100–1900:

centroid = Σ(ch × counts[ch], ch = 1100..1899) / Σ(counts[ch], ch = 1100..1899)

Центроид — взвешенное среднее. При пропорциональном дублировании чанков (USB-артефакт) числитель и знаменатель масштабируются одинаково — значение центроида не смещается. Этим объясняется иммунитет метода к артефактам чтения USB.

Baseline и обнаружение аномалии. Первые 5 окон формируют baseline; медиана их центроидов = baseline_median. Отклонение последующих окон:

Δ% = 100 × (centroid − baseline_median) / baseline_median

Статус OK при |Δ%| ≤ 2.0%; иначе — ANOMALY. Окна с нулевым центроидом (обрыв USB) пропускаются без записи в CSV.

Параметры запуска

stability_monitor_usb.py \
  --port <COM-port> --window 60 \
  --ch-start 1100   --ch-end 1900 \
  --anomaly-pct 2.0 --baseline-n 5 \
  --outdir <output-directory>

Артефакты счётчика: дублирование чанков USB

В процессе мониторинга фиксировались резкие скачки суммарного счёта (total_counts) в отдельных окнах — примерно 33 раза за первые 406 окон. Природа: порции гистограммы (чанки) при чтении через USB иногда считываются дважды, что удваивает счёт в соответствующих каналах этого окна.

Влияние на центроид: нулевое. Центроид — взвешенное среднее. При дублировании числитель (Σ ch·counts) и знаменатель (Σ counts) масштабируются пропорционально, отношение не меняется. Эмпирическое подтверждение: при ~33 флапах счётчика — 0 аномалий.

Ключевые фрагменты кода

Фреймер протокола shproto

class ShprotoParser:
    def __init__(self):
        self.buf = []; self.started = False; self.esc = False

    def feed(self, b: int):
        if b == 0xFE:
            self.buf = []; self.started = True; self.esc = False
            return None
        if not self.started:
            return None
        if b == 0xFD:
            self.esc = True
            return None
        if b == 0xA5:
            self.started = False
            if len(self.buf) >= 3 and crc16(bytes(self.buf)) == 0:
                cmd = self.buf[0]
                payload = bytes(self.buf[1:-2])
                return (cmd, payload)
            return None
        if self.esc:
            b = (~b) & 0xFF; self.esc = False
        self.buf.append(b)
        return None

Вычисление центроида

def centroid(histogram: list, ch_start: int, ch_end: int) -> float:
    total = sum(histogram[ch_start:ch_end])
    if total == 0:
        return 0.0
    return sum(c * histogram[c] for c in range(ch_start, ch_end)) / total

Классификация окна

delta_pct = 100 * (cen - baseline_median) / baseline_median
status = "ANOMALY" if abs(delta_pct) > args.anomaly_pct else "OK"

Импликации и следующие шаги

#STAB-1 → закрыт. Дрейф усиления детектора не является причиной аномалий #FW-16. Детектор AtomSpectra пригоден для длительных записей (24+ ч) без пересчёта энергетической калибровки.

#STAB-2 → следующий. Поскольку детектор стабилен, аномальный выброс ~8100 с в данных с платы (#FW-16) — артефакт тракта ESP32. Следующий тест: та же геометрия, источник и интервал, но сигнал идёт через плату. Добавлена телеметрия firmware-кольца (/api/devlog) для диагностики.

#WF-1 → код-фиксы ждут железного ретеста. Исправления гонки сегментации (рекурсия s_fs_flushed) уже влиты в main. Железный ретест водопада (≥1 ч) — после завершения обоих STAB-тестов.
