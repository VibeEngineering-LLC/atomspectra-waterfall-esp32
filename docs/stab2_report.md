## Cs-137 662 кэВ, через плату atomspectra-gw, 10.72 ч (9.40 ч чистого прогона после фикса)

**Дата:** 2026-07-03 → 2026-07-04
**Задачи:** #STAB-2 (task #29), #WF-1 (task #30)
**Статус:** ЗАВЕРШЁН. Фикс #WF-1 (`CONFIG_SPI_FLASH_AUTO_SUSPEND` выкл) подтверждён железным ретестом.

## 1. Цель

#STAB-1 показал: детектор AtomSpectra стабилен (0.023% дрейф за 9.6ч, USB напрямую).
Значит источник аномалий #FW-16 (выброс ~8100 с) — тракт ESP32, не детектор.

#STAB-2 повторяет измерение **через плату** (не напрямую по USB), с телеметрией
firmware-кольца (`/api/devlog`, `/api/waterfall/status`, `/api/system`), чтобы:
1. Привязать любые аномалии центроида к системным событиям платы (ребут, SNTP, ролловер сегмента).
2. Провалидировать фикс #WF-1 (отключение `CONFIG_SPI_FLASH_AUTO_SUSPEND`, commit `33fb4a4`) —
   до фикса плата уходила в краш-луп под нагрузкой записи водопада.

Тест начат на **предфиксовой** прошивке (репро краш-лупа), фикс залит по ходу теста,
далее — **9.40 ч чистого прогона** без единого ребута.

## 2. Схема эксперимента

```
[AtomSpectra детектор]
        │  USB (внутри платы)
        ▼
[ESP32-S3 atomspectra-gw]  ── WiFi ──►  [PC Windows 11]
  /api/spectrum.json                     stability_monitor_board.py
  /api/waterfall/status                        │
  /api/system                                  ├─ stability.csv     (1 строка/окно)
  /api/devlog?since=N                          ├─ telemetry.jsonl   (сырой снапшот wf/system)
                                                └─ devlog.txt        (лог платы)
```

Плата в схеме присутствует — сигнал измеряется через её HTTP API (не напрямую с детектора).
Метод изолирует именно тракт ESP32: USB-приём детектора → буфер → WiFi → HTTP.

## 3. Скрипт и методология

### Центроид — без изменений от #STAB-1

```
centroid = Σ(ch × counts[ch], ch=1100..1899) / Σ(counts[ch], ch=1100..1899)
Δ% = 100 × (centroid − baseline_median) / baseline_median
```

Первые 5 окон (до первого `BOARD_REBOOT`) — BASELINE. `baseline_median` = **1435.5321** кан.
Действует на весь прогон, включая период после ребутов (baseline не сбрасывается по `BOARD_REBOOT`,
т.к. это событие детектируется по регрессии `uptime_sec`, не по перезапуску скрипта-монитора).

### Телеметрия платы (новое относительно #STAB-1)

| Поле | Источник | Значение |
|---|---|---|
| `seg_dropped` | `/api/waterfall/status` | счётчик потерянных **flash-сегментов** водопада — целевая метрика фикса #WF-1 |
| `seg_count` | `/api/waterfall/status` | счётчик ролловеров сегмента (растёт → событие `SEG_ROLLOVER`) |
| `hist_drop` | `/api/spectrum.json` | счётчик пропущенных выборок **гистограммы** при HTTP/USB-фетче — метрика, независимая от `seg_dropped` |
| `uptime_sec` | `/api/system` | обнаружение `BOARD_REBOOT` (регрессия) и `SNTP_LIKELY` (uptime % 3600 около границы часа) |
| `/api/devlog?since=N` | инкрементальный | текстовый лог платы, событие `DEVLOG=n` при получении n новых строк |

`seg_dropped` и `hist_drop` — **разные счётчики разных подсистем**. `seg_dropped` считает потери
на пути «водопад → flash» (то, что чинил #WF-1). `hist_drop` считает потери на пути
«накопленная гистограмма спектра → HTTP-ответ `/api/spectrum.json`» — не связан с flash и не
входит в объём фикса #WF-1.

### Параметры запуска

```
python stability_monitor_board.py --host http://<board-host> --window 60
    --ch-start 1100 --ch-end 1900 --anomaly-pct 2.0 --baseline-n 5
    --outdir scripts\stability_board_20260703_234528
```

## 5. Артефакт: `hist_drop` — не путать с потерей сегментов

За Фазу B `hist_drop` вырос суммарно на **1239** (скорость ~2.20/мин). Это **не** регрессия
фикса #WF-1 — счётчик относится к другой подсистеме (см. §3): пропуски при фетче
гистограммы `/api/spectrum.json`, не при записи в flash. Аналогично артефакту дублирования
чанков USB в #STAB-1 — на центроид не влияет (взвешенное среднее иммунно к пропускам/
дублированию отдельных выборок), что подтверждается 0 ANOMALY при ненулевом `hist_drop`
во всех 526 окнах.

## 6. Артефакт: экспорт водопада ограничен ёмкостью flash-кольца (#FW-19)

При выгрузке n42-экспорта записанного водопада после теста обнаружено: файл содержит
только **256 строк** (~4.27 ч при кадансе ~60с/строка) вместо полных 9.40 ч чистого прогона.

Причина — `ring_capacity=256` (поле `/api/waterfall/status`): flash-кольцо физически хранит
только последние 256 строк. К концу теста накоплено `total_rows=568` строк — кольцо
перезаписало первые ~312 строк (~5.2 ч) новыми задолго до выгрузки экспорта.

**Это не регрессия #WF-1** — счётчик `seg_dropped` (потери именно на записи) остаётся 0,
т.е. запись в кольцо не теряет данные; ограничение — в объёме хранения, а не в надёжности
записи. Событие `RING_OVERFLOW` не сработало ни разу, т.к. его условие (`total_rows − flash_rows
≥ ring_capacity`) детектирует лаг записи, а не факт перезаписи кольца — для обнаружения
самого факта перезаписи нужна отдельная метрика (не реализована).

**Практическое следствие:** для записей длиннее ~4.25 ч нужен периодический pull данных
через `/api/waterfall` (клиент #REC-11) до заполнения кольца — иначе хвост записи не
восстановить постфактум. Зарегистрировано как **#FW-19** (задача #32, pending):
увеличить `ring_capacity` и/или задокументировать требование периодического pull.

## 8. Импликации

### #STAB-2 → закрыт
Тракт ESP32 (постфикс) не вносит дрейфа усиления сверх ожидаемого шума измерения
(0.50% макс. при пороге 2.0%, запас ≥4×). Аномалия #FW-16 (выброс ~8100 с) была следствием
краш-лупа/гонки записи флеша — устранена фиксом #WF-1, не связана с детектором или
нормальным трактом платы.

### #WF-1 → закрыт
Фикс (`CONFIG_SPI_FLASH_AUTO_SUSPEND` выкл, commit `33fb4a4`) подтверждён железным ретестом:
**0 ребутов, 0 потерянных сегментов за 9.40 ч** непрерывной записи водопада под реальной
нагрузкой (детектор + запись + WiFi-опрос раз в 60с). До фикса — ребут каждые ~15 мин.

### Новое: #FW-19 (pending, задача #32)
Обнаружена независимая (не #WF-1) капасити-проблема: экспорт водопада ограничен
`ring_capacity=256` строк (~4.25 ч). Требует отдельного решения — увеличение буфера
или дисциплина периодического pull во время длинных записей.
